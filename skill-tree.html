TL;DR: We're proposing to decompose the monolithic AI agent into 20 specialized skills across 6 domains. 70% of these skills already exist in the codebase as tools/features ‚Äî the work is decomposition and routing, not building from scratch. This board maps every skill, its data dependencies, and a 12-week migration path.

What is a "skill"?

  • A skill = a markdown playbook (500-1,000 lines) that gives the AI a specific persona, domain frameworks, data paths, pushback rules, and an output contract.
  • Think of it as: the difference between "GPT with bank data" and "a treasury analyst who's been at the company for 5 years."
  • Each skill reads structured data (its own + other skills'), applies domain frameworks, and produces both human-readable advice and machine-readable JSON.

How to use this board

  • Click any skill card to expand its full spec: persona, frameworks, data flow, pushback examples, output contract, and child skills.
  • Status badges tell you the migration effort: EXTRACT = already exists, just needs decomposition. NEW SKILL = needs to be built. CONNECT = exists as a separate service, needs API wrapper. ENHANCE = exists but needs upgrades. REDESIGN = needs rethinking.
  • Green code tags at the bottom of each card show which existing files/services map to this skill (e.g., bank_accounts_tool.rb).
  • üí¨ Notes button (bottom right) ‚Äî add your feedback, questions, or concerns. Pick your name and which skill the note is about. Press N to quick-open.
  • The Orchestrator section at the top shows how a single user question chains through multiple skills. This is the routing architecture.

What we need from you

  • Feasibility check: Do the "EXTRACT" skills map correctly to the existing codebase? Are we missing dependencies?
  • Priority input: Which domain should we decompose first? Treasury is the obvious choice, but engineering effort may differ.
  • Data layer: The cross-skill data flow proposes PostgreSQL JSONB. Does this fit your current architecture, or should we consider alternatives?
  • Router design: Intent classification ‚Üí skill chain. Should this be a lightweight LLM call, a rules engine, or something else?
  • Timeline reality check: The 12-week plan is aggressive. Where do you see the biggest risks?
üß† CFO Agent ‚Äî The Orchestrator
Routes intent, composes multi-skill responses, triggers proactive alerts. The only skill that reads from everything.
User: "Why is cash down $400K from last week?"
Step 1: Intent ‚Üí cash_position_change
Step 2: Activate ‚Üí Cash Analyst (position delta) + Transaction Classifier (large txns) + Variance Analyst (forecast vs actual)
Step 3: Synthesize ‚Üí "Three things: $180K Acme wire settled early, $150K payroll hit ahead of schedule, $70K FX conversion looks like a duplicate ‚Äî flagging for review."
Step 4: Proactive ‚Üí Risk Analyst triggered (duplicate FX) ‚Üí Insight Generator writes alert
Key principle: The router doesn't need to be smart. The skills are smart. The router just needs to know which skills to wake up and in what order. Intent taxonomy ‚Üí skill chain templates ‚Üí parallel execution ‚Üí synthesis.
